§1

Background: ERC-8183

ERC-8183, the Agentic Commerce Protocol, defines a job escrow with evaluator attestation for AI agent-to-agent commerce. The standard specifies three roles: a Client who creates the job and funds the escrow; a Provider who performs the work and submits a deliverable; and an Evaluator who attests completion or rejection. Jobs progress through six states: Open, Funded, Submitted, Completed, Rejected, and Expired.

The core interface surfaces eight functions covering the full job lifecycle. An optional IACPHook system provides the primary extension mechanism: each job may designate a hook contract that receives beforeAction and afterAction callbacks on every state transition, enabling validation logic, reputation updates, custom fee models, or dispute resolution — all without modifying the base protocol. The claimRefund function is deliberately not hookable as a safety guarantee.

Listing 1: ERC-8183 Core Interface
// ERC-8183 core functions
createJob(provider, evaluator, expiredAt, description, hook)
setProvider(jobId, provider, optParams)    // client assigns worker
setBudget(jobId, amount, optParams)        // client or provider negotiates
fund(jobId, expectedBudget, optParams)     // client escrows tokens
submit(jobId, deliverable, optParams)      // provider submits work
complete(jobId, reason, optParams)         // evaluator ✓ → pay provider
reject(jobId, reason, optParams)           // evaluator ✗ → refund client
claimRefund(jobId)                         // anyone, after expiry
The hooks system is the primary extension point. Each job can designate an IACPHook contract that receives beforeAction/afterAction callbacks on every state transition. Hooks can add validation, reputation updates, custom fee logic, or dispute resolution — all opt-in per job, with no modification to the base protocol.

The full specification is available at eips.ethereum.org/EIPS/eip-8183.

§2

The Problem

The ERC-8183 specification explicitly states: "No dispute resolution or arbitration; reject/expire is final." This is a deliberate design choice — keep the base protocol minimal. However, real agentic commerce requires considerably more nuance than a binary complete/reject outcome. The following limitations are inherent to the current design:

  • →
    Binary outcomes. complete() or reject() — nothing in between. A job that is 80% complete receives the same treatment as one that is 0% complete. There is no mechanism for graduated payment.
  • →
    No partial payouts. Late delivery, partial completion, and quality tiers are not representable. The spec has no mechanism for graduated fund distribution — the escrow either releases in full or refunds in full.
  • →
    No resubmission. A rejected provider cannot fix and resubmit. The job is terminated. A new job must be created from scratch, losing all context and requiring fresh funding.
  • →
    No dispute escalation. If the provider disagrees with the evaluator's rejection, no appeal path exists. The evaluator holds absolute power once the job reaches Submitted state.
  • →
    No AI adjudication standard. The evaluator is a single address — it could be an AI-powered contract, but the spec does not define how AI evaluation integrates with the job lifecycle, nor how multi-stage evaluation (e.g., technical review plus endorsement verification) should be composed.

Prior work on graduated penalty models in the Conditional Payment cross-border trade demo demonstrates six verdict tiers, 50–150bps penalties, and return-proof evaluation — all powered by GenLayer AI juries. The open question is how this should be standardized as a reusable hook extension to ERC-8183.

§4

Evaluation Criteria

The GenLayer AI jury evaluates every submission against six criteria. The evaluator contract aggregates two independent GenLayer oracle signals: (1) AI technical review against criteria 1–5, and (2) forum endorsement verification for criterion 6. The job remains in Submitted state while signals are collected. Both signals must pass for complete() to be called; if either rejects, reject() fires immediately.

    Design Memo

    Identify the specific limitations of ERC-8183's binary model. Propose a concrete extension using hooks and/or evaluator contracts. Explain how GenLayer AI adjudication integrates with the job lifecycle. Discuss tradeoffs explicitly.

  2. 2.

    Architecture Diagram

    Show AgenticCommerce as the base layer. Show how IACPHook and/or evaluator contracts extend it. Show the GenLayer bridge path. Identify which components live on which chain.

  3. 3.

    Judgment Model

    Go beyond binary complete/reject. Support at least two of: partial payout, penalty tiers, refund with deduction, resubmission window. Define verdict codes and fund distribution rules within ERC-8183's state machine.

  4. 4.

    Concrete Example Flow

    End-to-end walkthrough with specific values: token amounts, job IDs, timestamps. Show each ERC-8183 function call in sequence. Demonstrate a non-trivial outcome. Show the GenLayer AI jury step explicitly.

  5. 5.

    ERC-8183 Compatibility

    Do not modify AgenticCommerce or IERC8183. Use hooks and/or custom evaluator contracts only. Existing jobs without hooks must continue unchanged. The extension must be opt-in per job.

How to Submit

Prepare a proposal covering all six evaluation criteria, publish it at a publicly accessible URL, and interact with the on-chain contracts in sequence. Job ID is 0 on AgenticCommerce at 0x160b…abb1.

  1. Write your proposal addressing all six criteria: design memo, architecture diagram, judgment model, concrete example flow, ERC-8183 compatibility, and author endorsement plan.
  2. Publish the proposal at a publicly accessible URL — a GitHub repository, HackMD document, or hosted page.
  3. Post your proposal on the Ethereum Magicians ERC-8183 thread. Obtain a positive reply from any original EIP author (@dcrapis, @ai-virtual-b, @twx-virtuals, or @Zuhwa).
