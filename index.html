The primitive

What is ERC-8183?

A minimal escrow with an external evaluator

ERC-8183 defines the simplest possible pattern: lock funds, submit something for evaluation, get a verdict, distribute funds. It deliberately says nothing about what is evaluated or how the verdict maps to payouts — that's what extensions are for. 

interface IERC8183 {
  // Deposit funds into escrow
  function deposit(address token, uint256 amount) external payable;

  // Submit data for evaluation → returns unique submission ID
  function submit(bytes calldata data) external returns (bytes32);

  // Evaluator renders a verdict on a submission
  function resolve(bytes32 submissionId, uint8 verdict, bytes calldata data) external;

  // Views
  function status() external view returns (uint8);
  function evaluator() external view returns (address);
  function depositor() external view returns (address);
  function escrowToken() external view returns (address);
  function escrowBalance() external view returns (uint256);
}

Three operations. Five views. That's the entire standard. Everything else is an extension.

Evaluation criteria

What your proposal must include

The GenLayer AI jury evaluates every submission against these five criteria. All five must pass for the bounty to be awarded.

1

Design memo

A clear document stating the problem, your proposed extension architecture, rationale, and tradeoffs. Professional engineering quality.

2

Architecture diagram

A visual overview showing how your extension relates to the ERC-8183 base. Contracts, components, data flow. ASCII, Mermaid, or image — all count.

3

Judgment model

How evaluator verdicts map to outcomes. Must support at least: partial payout, full refund, penalty, or resubmission. Go beyond binary accept/reject.

4

Concrete example flow

An end-to-end walkthrough with specific values — amounts, addresses, timestamps. Show each step from deposit through at least one non-trivial resolution.

5

ERC-8183 preservation

Your extension must NOT modify IERC8183 itself. Layer on top via inheritance, composition, or wrapping. Existing ERC-8183 contracts must remain valid.

The relay picks up the event and deploys a ProposalEvaluator contract on GenLayer. Multiple AI validators independently fetch your proposal, evaluate it against the five criteria, and reach consensus.

Verdict delivered via bridge

The GenLayer verdict (ACCEPT or REJECT) is encoded and sent through LayerZero back to Base Sepolia. The InternetCourtFactory delivers it to the bounty contract via resolve().

🏆

Prize released or bounty stays open

ACCEPT → The full 50,000 PEN prize transfers to the submitter's address automatically.
REJECT → The bounty remains open. You receive specific feedback from the jury and can resubmit an improved proposal.

How to submit a proposal

Prepare your proposal, publish it at a URL the AI jury can fetch, and submit it on-chain.

  • Write your proposal covering all five criteria (design memo, architecture, judgment model, example flow, ERC-8183 preservation)
  • Publish it at a publicly accessible URL — a GitHub repository, HackMD doc, or hosted page
  • Call submit(abi.encode(proposalUrl)) on the BountyEscrow8183 contract on Base Sepolia

Context

Why extend ERC-8183?

The base primitive — escrow + evaluate + resolve — handles binary outcomes well. But real commerce needs more nuance:

→

Partial payouts. A shipment arrives 3 days late — not a total failure, but not on time. The penalty should be proportional.

→

Graduated penalties. Different delay buckets map to different penalty tiers — 50bps, 100bps, 150bps — not all-or-nothing.

→

Return & resubmission. When goods are rejected, the return proof itself needs evaluation before any refund.

→

Multi-stage adjudication. Some disputes need sequential evaluation phases — initial verdict, appeal window, final resolution.

We've built this exact pattern in the Conditional Payment demo — 6 graduated verdicts, penalty tiers from 50–150bps, a return-proof timeout mechanism, all evaluated by GenLayer AI juries. The question is: how should this be standardized as a reusable extension to ERC-8183?
