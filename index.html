The standard

What is ERC-8183?

Agentic Commerce Protocol

ERC-8183 defines a job escrow with evaluator attestation for AI agent commerce. Three roles, six states, and an optional hooks system for extensibility. Read the full EIP →

Three roles

  • Client — creates job, funds escrow
  • Provider — does the work, submits deliverable
  • Evaluator — attests completion or rejection

Six states

  • Open → Funded → Submitted
  • → Completed (payment released)
  • → Rejected (refund to client)
  • → Expired (timeout refund)
// ERC-8183 core functions
createJob(provider, evaluator, expiredAt, description, hook)
setProvider(jobId, provider, optParams)    // client assigns worker
setBudget(jobId, amount, optParams)        // client or provider negotiates
fund(jobId, expectedBudget, optParams)     // client escrows tokens
submit(jobId, deliverable, optParams)      // provider submits work
complete(jobId, reason, optParams)          // evaluator ✓ → pay provider
reject(jobId, reason, optParams)            // evaluator ✗ → refund client
claimRefund(jobId)                          // anyone, after expiry
🔑

The hooks system is the extension point. Each job can have an IACPHook — a contract that receives beforeAction/afterAction callbacks on every state transition. Hooks can add validation, reputation updates, custom fee logic, or dispute resolution. claimRefund is deliberately not hookable — safety guarantee.

The problem

What ERC-8183 doesn't do

The spec explicitly states: "No dispute resolution or arbitration; reject/expire is final." This is a deliberate design choice — keep the base minimal. But real commerce needs more nuance:

→

Binary outcomes. complete() or reject() — nothing in between. A job that's 80% done gets the same treatment as one that's 0% done.

→

No partial payouts. Late delivery, partial completion, quality tiers — the spec has no mechanism for graduated fund distribution.

→

No resubmission. A rejected provider can't fix and resubmit. The job is dead. A new job must be created from scratch.

→

No dispute escalation. If the provider disagrees with the evaluator's rejection, there's no appeal path. The evaluator has absolute power once the job is Submitted.

→

No AI adjudication. The evaluator is a single address. It could be an AI-powered contract — but the spec doesn't define how AI evaluation integrates with the lifecycle.

We've built graduated penalty models in the Conditional Payment demo — 6 verdict tiers, 50–150bps penalties, return-proof evaluation — all powered by GenLayer AI juries. The question is: how should this be standardized as a reusable hook extension to ERC-8183?

Evaluation criteria

What your proposal must include

The GenLayer AI jury evaluates every submission against five criteria. All five must pass for complete() to be called.

1

Design memo

Identify the specific limitations of ERC-8183's binary model. Propose a concrete extension using hooks and/or evaluator contracts. Explain how GenLayer AI adjudication integrates with the job lifecycle. Discuss tradeoffs.

2

Architecture diagram

Show AgenticCommerce as the base layer. Show how IACPHook and/or evaluator contracts extend it. Show the GenLayer bridge path. Identify which components live on which chain.

3

Judgment model

Go beyond binary complete/reject. Support at least two of: partial payout, penalty tiers, refund with deduction, resubmission. Define verdict codes and fund distribution rules within ERC-8183's state machine.

4

Concrete example flow

End-to-end walkthrough with specific values: token amounts, job IDs, timestamps. Show each ERC-8183 function call in sequence. Demonstrate a non-trivial outcome. Show the GenLayer AI jury step.

5

ERC-8183 compatibility

Do NOT modify AgenticCommerce or IERC8183. Use hooks and/or custom evaluator contracts. Existing jobs without hooks must continue unchanged. The extension is opt-in per job.

Bounty lifecycle

How this bounty works

This bounty is itself an ERC-8183 job. The full lifecycle follows the spec exactly.

📝

Job created (Open)

Client calls createJob(provider=0, evaluator=GenLayerEvaluator, hook=CourtAwareHook). Provider is open — anyone can claim it.

🙋

Provider assigned

When you want to submit, the client calls setProvider(jobId, yourAddress) to assign you. Then fund(jobId, budget) escrows the 50,000 PEN prize.

Open → Funded
📄

Register + Submit

You call registerProposal(jobId, url, title) on the CourtAwareHook (metadata), then submit(jobId, keccak256(url)) on AgenticCommerce.

Funded → Submitted
🧠

GenLayer AI jury evaluates

The relay deploys a ProposalEvaluator on GenLayer. Multiple AI validators independently fetch your proposal, evaluate against the five criteria, and reach consensus.

⚖️

Verdict delivered via bridge

The GenLayer verdict is bridged via LayerZero to the GenLayerEvaluator contract on Base Sepolia. It calls either complete() or reject() on the job.

🏆

Outcome

ACCEPT → complete(): 50,000 PEN released to your address.
REJECT → reject(): escrow refunded to client. New job created for next attempt.

Submitted → Completed | Rejected
